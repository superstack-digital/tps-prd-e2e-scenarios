Overview & Purpose

What this document is and who it is for

Purpose

This document defines the complete end-to-end test plan for The Padel Society tournament system. It covers all tournament formats available when an organizer creates a tournament event. Each scenario specifies preconditions, user steps, and exact assertions so QA engineers can automate or manually validate the feature.

Audience

  • QA Engineers running E2E or manual tests
  • Product Managers validating behaviour before release
  • Engineers reviewing expected system contracts
  • Organizers onboarding onto new tournament formats

Realistic Player Counts

Most clubs run 3–5 courts. A typical event has around 10 players — sometimes as few as 6, sometimes up to 16. Below 6 players a tournament loses its purpose (4 players is just a regular match). Scenarios use these real-world sizes. Hard boundary tests (e.g. exact minimum validation) are labelled explicitly as edge cases.

Test Execution Approach

  • Automated E2E: Playwright against staging device/simulator
  • Manual validation: organizer role on real event
  • Unit tests cover pairing algorithms separately
  • Feature flags tested in isolation before rollout

Known Bugs — Test Against These

Active defects found in code review. Scenarios referencing these include a warning note.

RC-1
Standings fetched by points instead of rank — wrong Mexicano pairings on ties

File: event-tournament-v3/index.ts · Line 257

Root Cause: .order('points', {ascending: false}) — two players with equal points but different tiebreaker ranks produce incorrect next-round pairings.

Fix: Replace with .order('rank', {ascending: true})

Affected formats: Classic Mexicano, Team Mexicano

RC-2
Future rounds deleted before completeness check — premature round wipe

File: event-tournament-v3/index.ts · Lines 228–234

Root Cause: DELETE fires before checking if all matches in the current round are complete. If only one match is submitted, future rounds are wiped prematurely.

Fix: Move DELETE inside the if (isComplete) block

Affected formats: Classic Mexicano, Team Mexicano

RC-3
Score stored as sets won regardless of scoring mode — wrong standings in points mode

File: event-tournament-v3/index.ts · Lines 854–855

Root Cause: storedTeam1Score = team1SetsWon always — even when match is in points mode (not multi-set).

Fix: isMultiSet ? team1SetsWon : team1Score

Affected formats: All formats using KOC score handler

RC-4
KOC standings fetch missing rank ordering — potential wrong court assignments

File: event-tournament-v3/index.ts · Lines 474–478

Root Cause: KOC standings fetch does not include .order('rank', {ascending: true}), same class of bug as RC-1.

Fix: Add .order('rank', {ascending: true}) to standings query

Affected formats: King of Courts, Team King of Courts

RC-5
No partner/matchup history check in Mexicano pairing — repeats allowed

File: MexicanoTournamentService.ts · Lines 1123–1160 (createMatchesFromStandingsStatic)

Root Cause: Always applies rank 1+3 vs 2+4 formula with no check against partner or matchup history.

Fix: Try 3 pairing configurations per group of 4, pick the first that avoids last-round partner and repeated matchup.

Affected formats: Classic Mexicano, Team Mexicano

AME-001
Create Americano with 16 players across 4 courts
Happy Path · Setup

Preconditions

  • Organizer is logged in and has an active event
  • Event has 4 courts registered
  • 16 participants confirmed on the event
  • No active tournament on this event

Steps

  1. Navigate to event → Tournament tab → "Create Tournament"
  2. Select tournament type: Americano
  3. Select all 16 players from participant list
  4. Select 4 courts, set maxPoints to 16
  5. Tap "Generate Tournament"

Assertions

  • Tournament created with type americano, status pending
  • Round 1 contains exactly 4 matches (4 courts × 1 match each)
  • All 16 players distributed across 4 matches — each player in exactly one team slot
  • Each match has team1Player1, team1Player2, team2Player1, team2Player2 populated
  • Match scores are null, status pending
  • All matches have maxPoints = 16
  • Standings table shows 16 entries, all at 0 points, 0 wins, 0 losses
  • No usersSittingOutInRound entries exist

Edge Cases

  • 20 players, 4 courts (not divisible by 4 per court) → creation fails with player count validation
  • 12 players, 3 courts → 3 matches, all 12 play — valid
  • 0 courts selected → creation fails
AME-002
Submit score on one court — standings update correctly
Happy Path · Scoring

Preconditions

  • Americano with 16 players, 4 courts, round 1 active (4 matches pending)

Steps

  1. Open match on Court 1: Team A (Player 1 + Player 2) vs Team B (Player 3 + Player 4)
  2. Enter Team A score: 16, Team B score: 9
  3. Tap "Submit Score"
  4. Navigate to Standings tab

Assertions

  • Court 1 match status changes to completed
  • Players 1 and 2 each: wins=1, points=2 (or configured win value), pointsFor=16, pointsAgainst=9
  • Players 3 and 4 each: losses=1, points=0, pointsFor=9, pointsAgainst=16
  • Players 1 and 2 ranked above 3 and 4 in standings
  • Players on courts 2, 3, 4 still show 0 points (matches still pending)
  • Round 1 shows as "3 of 4 matches remaining"

Edge Cases

  • Tied score (8–8) → both teams receive ties=1; standings points equal; tiebreaker by differential = 0
  • Score exceeds maxPoints (17–5 when maxPoints=16) → validation error
  • Non-organizer submit attempt → 403 / hidden submit button
AME-003
Round 2 auto-generates after all round 1 matches complete
Happy Path · Rotation

Preconditions

  • Americano with 16 players, 4 courts, all 4 round 1 matches pending

Steps

  1. Submit scores for all 4 round 1 matches
  2. Observe whether round 2 is generated
  3. Compare round 2 player assignments to round 1

Assertions

  • Round 2 appears automatically after the 4th round 1 score is submitted
  • Round 2 has exactly 4 matches (all 16 players still participate)
  • No player appears in more than one match in round 2
  • All 16 players assigned across 4 matches
  • At least some courts have different 4-player groupings vs round 1 (pairing is random)
  • Round 1 remains visible in history

  2. Inspect every round's match assignments

Assertions

  • Every player appears in exactly 1 match in every round, across all rounds
  • usersSittingOutInRound is always empty for Americano
  • Total match slots per round = number of players (no idle players)
AME-005
Final standings: point differential breaks equal win counts
Edge Case · Tiebreaker

Preconditions

  • Americano, 4 rounds completed. Players A and B both have 2 wins, 2 losses (equal points).
  • Player A: pointsFor=58, pointsAgainst=42 (+16 differential)
  • Player B: pointsFor=52, pointsAgainst=46 (+6 differential)

Steps

  1. Complete all rounds with the above scores
  2. Check standings order

Assertions

  • Player A ranked above Player B (better differential)
  • Tiebreaker applied: 1) total standings points, 2) pointsFor − pointsAgainst
  • Differential value shown in standings UI

MEX-003
Round 2 pairing: rank 1+3 vs rank 2+4 per group of 4
Happy Path · Pairing Logic

Preconditions

  • Mexicano, 16 players, 4 courts. After round 1, clear ranking established.
  • Top 8 players ranked 1–8, bottom 8 ranked 9–16

Steps

  1. Complete round 1 with distinct score differences to produce clear ranking
  2. Observe round 2 pairings on all 4 courts

Assertions

  • Court 1 (top group): rank 1 + rank 3 vs rank 2 + rank 4
  • Court 2 (second group): rank 5 + rank 7 vs rank 6 + rank 8
  • Court 3: rank 9 + rank 11 vs rank 10 + rank 12
  • Court 4 (bottom group): rank 13 + rank 15 vs rank 14 + rank 16
  • NOT rank 1+2 vs 3+4 (which would separate top ranked players)
⚠ RC-1: Standings ordered by points not rank — ties with different tiebreaker ranks may produce wrong pairings. Fix: .order('rank', ascending: true).

MEX-006
Partner history: system attempts to avoid same partner in consecutive rounds
Edge Case · Partner Rotation

Preconditions

  • Mexicano, 16 players. Round 1 complete: Player A partnered with Player C.

Steps

  2. Observe round 2 — check whether A and C are partners again

Assertions

  • If rankings allow an alternative, A and C are NOT paired in round 2
  • If no alternative is mathematically possible, repeat is allowed
⚠ RC-5: createMatchesFromStandingsStatic() currently has no partner history check. Repeats occur freely. Fix pending: try 3 configurations, pick first that avoids last-round partner.
MEX-007

MEX-010
Partial round score does NOT reassign partners or delete future rounds
Regression · RC-1 / RC-6

Preconditions

  • Classic Mexicano, 16 players, 4 courts, round 1 generated
  • Round 2 placeholder already exists in DB (generated at tournament creation)
  • Record the partner assignment for every player in round 1 before submitting any score

Steps

  1. Submit the score for Court 1 only (1 of 4 courts scored)
  2. Immediately check: round 1 partner assignments for all 4 courts
  3. Immediately check: round 2 data still exists in DB
  4. Submit scores for Courts 2 and 3 (3 of 4 courts scored)
  5. Repeat check: round 1 assignments unchanged, round 2 still present
  6. Submit the final score for Court 4 (round complete)
  7. Now check: round 2 regenerated from updated standings

Assertions

  • After step 2: all round 1 partner assignments identical to pre-score snapshot — no player moved
  • After step 2: round 2 DB rows still present — not deleted
  • After step 5: same — partner assignments and round 2 untouched after 3rd score
  • After step 7: round 2 regenerated from updated standings (this is expected and correct)
  • At no point between steps 1–5 does any player's partner_id change
  • At no point between steps 1–5 are future round rows deleted from event_rounds
⚠ RC-1 + RC-6: Without the completeness guard, each partial score triggers generateNextRound() + deleteRoundsAfter() immediately — reassigning partners and wiping future rounds mid-round. This test catches both regressions in a single scenario.
